Data Flow

Core Layers

Job Queue Layer
  • 5 BullMQ queues — each with independent retry strategies
  • Webhook deduplication via stable jobId
  • Exponential backoff (5s → 10s → 20s for ticket sync)
  • Bull Board UI at /api/admin/queues
Redis Caching
  • Raw string for the 20–50MB active tickets payload (zero parse cost)
  • Hash for O(1) per-ticket lookups
  • Distributed lock (SET NX EX) to prevent cache stampede
  • Non-blocking SCAN for invalidation (not KEYS *)
MongoDB
  • Denormalized AnalyticsTicket schema — owner/cohort pre-resolved
  • Cursor-based pagination (no .skip() at scale)
  • Compound indexes aligned to query + sort patterns
  • TTL indexes for auto-cleanup (remarks: 30d, cache: 24h)
Real-Time Layer
  • Redis Pub/Sub bridges worker → API process
  • Socket.IO pushes SYNC_PROGRESS + DATA_UPDATED
  • Separate pub/sub clients (Redis protocol requirement)
  • Hybrid process mode — split via NODE_ROLE flag

Key Design Decisions

Decision Why
BullMQ over direct async calls Rate limit compliance, CPU isolation, automatic retry — 70 users can't each trigger 70 DevRev API calls
5 separate queues A 15-min historical sync must not block a 30-sec ticket refresh — independent queues, independent failure profiles
Raw string caching for tickets Avoids 250ms of JSON.parse + stringify and ~40MB memory alloc per request on a 20–50MB payload
Cursor pagination over .skip() .skip(10000) scans and discards 10k docs; cursor seeks directly via index — O(1) vs O(N)
Denormalized schema Pre-resolving owner, cohort, and NOC status at write time makes analytics queries instant — no N+1 lookups
JWT + API Keys JWT for browser sessions (Google OAuth), scoped API keys for machine-to-machine consumers — hashed with HMAC-SHA256
